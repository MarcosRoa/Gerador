🧠 Loterias V.7.0 API

Documentação da API REST

🗄️ API - Endpoints

API centralizada via Vercel Functions com TypeScript

📡 Endpoints Disponíveis

GET /api/user/credits
Retorna o saldo do usuário logado
Headers: Authorization: Bearer {firebase_token}, X-User-Id, X-User-Email
Response: { credits: number, isPro: boolean }
GET /api/user/history
Retorna histórico de palpites
Response: { history: [], count: number }
GET /api/pro/status
Retorna status do plano PRO
Response: { isPro: boolean, daysLeft: number }
POST /api/generate
Gera jogos e debita créditos RPC Atômica
Body: { lottery, quantity, mode, extraNumbers }
Response: { games, creditsRemaining, gameId, idempotencyKey }
POST /api/payments/create
Cria cobrança PIX (simulação ou real)
Body: { amount, userId }
Response: { qrCode, copyPaste, paymentLink }
POST /api/payments/webhook
Webhook do Mercado Pago (PIX)
Body: payload do Mercado Pago
⚠️ Requer verificação de assinatura
GET /api/health
Health check para monitoramento
Response: { status, version, checks }

🔐 Autenticação

Todas as requisições (exceto webhook) requerem:

Authorization: Bearer {firebase_id_token}
X-User-Id: {firebase_uid}
X-User-Email: {user_email}
X-User-Name: {user_name}

⚡ RPC Atômicas (Supabase)

Transações TUDO ou NADA para operações críticas:

-- Geração de jogos (débito + histórico)
SELECT * FROM generate_game(
    p_user_uid, p_lottery, p_games, p_cost, p_mode, p_extra_numbers, p_idempotency_key
);

-- Adição de créditos (compra PIX)
SELECT * FROM add_credits(
    p_user_uid, p_amount, p_payment_id, p_idempotency_key
);

🛡️ Proteções

  • Rate Limiting: 30 req/min (padrão), 5 req/10s (generate)
  • Idempotência: Evita duplicação via idempotency_key
  • Pessimistic Locking: SELECT FOR UPDATE nas RPCs
  • Validação: Zod schemas em todos os endpoints
